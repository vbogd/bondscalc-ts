# План: данные MOEX для поиска и калькулятора

## Цель

Убрать дублирование и рассинхронизацию кэшей данных MOEX, сохранив общий
snapshot primary board для поиска и калькулятора. Затем исправить расчёт
валютных облигаций так, чтобы цена бралась с основной торговой доски, а все
денежные потоки и НКД были выражены в валюте номинала.

## Общие правила

- Все HTTP-запросы к MOEX ISS живут в одном модуле
  `src/shared/api/moex/client.ts`; новых запросов к MOEX вне этого модуля не добавлять.
- У каждой экспортируемой функции в `client.ts` есть короткий комментарий на
  английском: какие данные она получает, для какого сценария нужна и какие
  допущения делает.
- Нормализация ответов MOEX остаётся в `bonds.ts`, типы — в `types.ts`.
  Компоненты страниц получают уже нормализованные данные и сами HTTP не
  вызывают.
- Обычные unit-тесты используют mock-ответы. Интеграционные тесты MOEX
  запускаются отдельной командой.

## 1. Единый React Query-кэш primary-board snapshot

### Проблема

Сейчас полный snapshot `securities.json` кэшируется в API-клиенте 60 секунд,
а React Query считает данные устаревшими через 30 секунд. При повторном
запросе через 30–60 секунд React Query может отметить старый snapshot как
только что обновлённый.

### Работы

1. Вынести загрузку полного snapshot в чистую функцию API-клиента без
   внутреннего кэша и без TTL.
2. Создать общий React Query query key для snapshot primary board, например
   `['moex', 'primary-bond-snapshot']`, со `staleTime: 30_000`.
3. Перевести SearchPage на этот query:
   - snapshot загружается один раз;
   - поиск и glob-фильтрация выполняются локально по уже полученному списку;
   - изменение текста поиска не создаёт новый MOEX-запрос.
4. Перевести CalculatorPage на тот же query:
   - при переходе из поиска используются уже загруженные базовые данные;
   - при прямом открытии калькулятора загружается тот же shared snapshot;
   - отдельную fallback-ветку с другим способом загрузки базовой информации не
     добавлять.
5. Удалить внутренний Promise-кэш и 60-секундный TTL из API-клиента.
6. Сохранить существующее поведение обновления формы: фоновые данные не
   перезаписывают изменённые пользователем поля без явного действия.

### Критерии готовности

- Поиск и калькулятор используют один React Query key и один срок свежести —
  30 секунд.
- Открытие калькулятора из поиска не создаёт второй запрос полного
  `securities.json`.
- Прямая ссылка на калькулятор использует тот же snapshot, без отдельной
  логики загрузки.
- После истечения 30 секунд параллельные потребители получают результат одного
  дедуплицированного запроса.
- Есть тесты на общий кэш, отсутствие сетевого обращения при локальной
  фильтрации и корректное обновление после устаревания данных.

## 2. Минимизация трафика MOEX ISS

### Цель

Каждый запрос к MOEX должен получать только блоки и колонки, которые реально
используются приложением. Полный snapshot primary board остаётся единственным
осознанным исключением: он нужен для локального поиска по всем облигациям.

### Работы

1. Для каждого запроса в `client.ts` зафиксировать назначение, используемые
   блоки ISS и необходимый набор колонок.
2. Добавить `iss.only` в запросы, которым не нужны все блоки ответа.
3. Добавить параметр `*.columns` для каждого запрашиваемого блока и исключить
   поля, которые не используются нормализаторами или страницами.
4. Сократить исторический запрос до действительно используемых полей. Сейчас
   калькулятор использует исторические НКД и номинал; не запрашивать купонные
   поля, пока они не нужны расчёту.
5. Для запросов данных одного выпуска оставить только необходимые market-data
   поля по доскам: идентификатор, валюту расчётов, признаки primary/торгов,
   цену, НКД и показатели ликвидности, которые показывает интерфейс.
6. Обновить unit-тесты API-клиента: они должны проверять точные URL, блоки и
   списки колонок, чтобы лишние данные не вернулись при дальнейших изменениях.

### Критерии готовности

- Все точечные запросы MOEX содержат минимальные `iss.only` и `*.columns`.
- В коде нет чтения полей, которые не были явно запрошены.
- Интеграционные тесты подтверждают, что ограниченные ответы по-прежнему
  нормализуются для реальных выпусков.
- Полный snapshot используется только как shared data source поиска и
  калькулятора по пункту 1.

## 3. Валютные облигации: разделение торговых и денежных данных

### Проблема

У валютных выпусков основная торговая доска может рассчитываться в RUB, а
номинал и купоны заданы в USD, EUR или CNY. Например, для `RU000A107B43`
цена и объём берутся с primary board `TQCB`, но НКД этой доски указан в RUB;
его нельзя передавать в долларовый расчёт вместе с номиналом и купонами в USD.

### Правило данных

- Цена, предыдущая цена, объём и число сделок берутся с primary board: там
  проходит реальная ликвидность.
- Номинал, купоны, амортизации, погашение, НКД и результаты расчёта выражаются
  в `FACEUNIT` облигации.
- Для RUB допускается эквивалентность кодов `SUR` и `RUB`.
- Не подменять НКД в валюте номинала рублёвым НКД и не выполнять неявную
  конвертацию по курсу.

### Работы

1. Расширить нормализованную модель досок: хранить идентификатор доски, признак
   primary, валюту расчётов и только нужные market-data поля.
2. Сделать небольшой запрос данных выбранного выпуска по всем его доскам.
   Ограничить блоки и колонки MOEX только значениями, нужными калькулятору.
3. Сохранить primary board источником цены и ликвидности.
4. Для НКД выбрать данные доски, у которой `CURRENCYID` совпадает с
   `FACEUNIT`, без хардкода названий досок (`TQOD`, `TQOE`, `TQOY`).
5. Если MOEX не возвращает НКД в валюте номинала, не показывать ложную
   доходность. Отдельно согласовать безопасный fallback: точный расчёт НКД по
   графику или явное сообщение о невозможности расчёта.
6. Применить то же правило к прошлой дате покупки: исторический НКД не должен
   смешивать валюту primary board с валютой cash flow.
7. Обновить отображение детализации и итогов: денежные значения калькулятора
   показываются в `FACEUNIT`, а не в `CURRENCYID` primary board.

### Критерии готовности

- `RU000A107B43` использует цену primary board и НКД в USD; доходность не
  становится отрицательной из-за рублёвого НКД.
- Имеются unit-тесты для USD, EUR и CNY без зависимости от названий досок.
- Есть регрессия для обычной рублёвой облигации.
- Нет неявной конвертации RUB/USD/EUR/CNY и нет смешения денежных величин
  разных валют в одном расчёте.
- Запросы к MOEX остаются только в `client.ts`, а экспортируемые функции
  содержат английские комментарии.
