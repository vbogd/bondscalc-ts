# План работ: этап 6

Этап 6 из `docs/PROJECT_PLAN.md` делим на два рабочих подэтапа:

1. Подключение минимальной UI-базы `shadcn/ui`.
2. Дизайн-полировка, мобильная проверка и подготовка static deploy.

Цель - не переписать приложение ради библиотеки, а использовать `shadcn/ui` как набор проверенных accessible-примитивов. Продуктовый дизайн, доменные компоненты и финансовая логика остаются внутри проекта.

## Этап 6.1: подключение `shadcn/ui`

### Цель

Добавить минимальную UI-базу без изменения продуктового поведения.

### Работы

1. Подключить `shadcn/ui` к существующему Vite + React + Tailwind проекту.
2. Настроить `components.json`, path aliases и helper `cn`, если CLI добавит их.
3. Настроить базовые CSS variables так, чтобы они не конфликтовали с текущим интерфейсом.
4. Добавить только нужные компоненты:
   - `button`;
   - `input`;
   - `badge`;
   - `tooltip`;
   - `dropdown-menu`;
   - `toggle-group`.
5. Не подключать на этом этапе:
   - `card`;
   - `table`;
   - `sidebar`;
   - `form`;
   - `calendar`;
   - `data-table`.
6. Сделать минимальную замену там, где это безопасно:
   - кнопка назад;
   - кнопка внешних ссылок;
   - tooltip в результатах;
   - переключатель режима, если `toggle-group` ложится без усложнения.
7. Не менять:
   - финансовую логику;
   - MOEX-клиент;
   - структуру экранов;
   - пользовательские сценарии.

### Проверки

```bash
npm run test
npm run build
```

### Критерий готовности

- Проект собирается.
- `shadcn/ui` подключен.
- Добавлены только нужные UI-примитивы.
- Поведение приложения не изменилось.
- Нет крупных дизайн-изменений, которые усложняют диагностику подключения.

### Suggested commit message

```text
Add minimal shadcn ui primitives
```

## Этап 6.2: дизайн-полировка и static deploy

### Цель

Привести интерфейс к направлению из `DESIGN.md`, проверить мобильные ширины и подготовить финальную статическую сборку.

### Дизайн-токены

Настроить токены в CSS/Tailwind-слое под `DESIGN.md`:

- primary: `#0052ff`;
- primary active: `#003ecc`;
- ink: `#0a0b0d`;
- body: `#5b616e`;
- muted: `#7c828a`;
- muted soft: `#a8acb3`;
- canvas: `#ffffff`;
- surface soft: `#f7f7f7`;
- surface strong: `#eef0f3`;
- border: `#dee1e6`;
- semantic up: `#05b169`;
- semantic down: `#cf202f`.

Правила цвета:

- Coinbase Blue использовать редко: primary action, focus, selected state.
- Зеленый и красный использовать только для финансового смысла.
- Не вводить дополнительный брендовый цвет.
- Убрать лишние emerald/amber акценты, если они не несут смысл риска или финансовой семантики.

### Типографика

1. Использовать Inter как замену Coinbase Sans/Display.
2. Заголовки держать спокойными по весу, без агрессивного bold.
3. Все числовые значения показывать моноширинным шрифтом:
   - цены;
   - проценты;
   - деньги;
   - доходности;
   - сроки;
   - итоговые расчетные значения.

### Доменные UI-компоненты

Поверх `shadcn/ui` собрать небольшие проектные компоненты:

- `SearchInput`;
- `BondBadge`;
- `ModeToggle`;
- `InputField`;
- `ResultPanel`;
- `ResultRow`;
- `StateMessage`.

Эти компоненты должны отражать домен приложения, а не просто прокидывать props в `shadcn/ui`.

### Экран поиска

1. Привести search input к pill-паттерну:
   - `rounded-full`;
   - мягкий серый фон;
   - синий focus ring;
   - высота не меньше 44px.
2. Результаты оформить как строгие financial rows:
   - спокойные разделители;
   - понятная иерархия названия, ISIN, дат и показателей;
   - числа моноширинным шрифтом.
3. Бейджи уровня листинга и оферты сделать спокойными pill-badges.
4. Проверить состояния:
   - меньше 3 символов;
   - loading;
   - error;
   - empty;
   - список результатов.
5. Проверить touch targets на мобильном экране.

### Экран калькулятора

1. Аккуратно разделить блоки:
   - шапка бумаги;
   - параметры выпуска;
   - покупка;
   - сценарий выхода;
   - результаты;
   - детализация.
2. Привести inputs к единому виду:
   - белый фон;
   - hairline border;
   - blue focus;
   - достаточная высота для touch.
3. Переключатель `Погашение / Оферта / Продажа` сделать как segmented control.
4. Результаты сделать главным визуальным блоком:
   - ключевые показатели крупнее;
   - все числа моноширинным шрифтом;
   - прибыль/убыток выделять цветом только по смыслу.
5. Детализацию сделать компактной и читаемой.
6. Проверить tooltip:
   - не выходит за экран на 320-375px;
   - доступен с клавиатуры;
   - не перекрывает критичные элементы.

### Мобильная проверка

Обязательные ширины:

- `320px`;
- `375px`;
- `390px`;
- `430px`;
- `640px`.

Проверить оба экрана:

- нет горизонтального скролла;
- длинные названия облигаций не ломают layout;
- значения не перекрывают подписи;
- кнопки и поля не скачут по высоте;
- date input выглядит приемлемо;
- меню внешних ссылок не уезжает за экран;
- основные действия имеют touch target не меньше 44px.

### Playwright

Решение принимается после ручной мобильной проверки.

Добавлять Playwright, если есть риск визуальных или навигационных регрессий, которые легко закрепить smoke-тестами.

Минимальный набор, если Playwright добавляется:

1. Стартовая страница открывается на mobile viewport.
2. Ввод менее 3 символов не запускает поиск.
3. Состояние поиска отображается корректно.
4. Страница калькулятора открывается с замоканными MOEX-ответами.
5. Ключевые блоки не перекрываются на `375x812`.

Реальные MOEX-запросы не делать частью e2e. Для них уже есть отдельные integration-тесты.

### Static deploy

1. Проверить production build:

```bash
npm run build
```

2. Проверить preview:

```bash
npm run preview
```

3. Обновить README короткой секцией по static hosting.
4. Если выбран GitHub Pages:
   - настроить `base` в Vite;
   - учесть SPA fallback для прямого открытия `/bond/:secid`.
5. Если выбран Cloudflare Pages, Vercel static hosting или похожий сервис:
   - описать build command;
   - описать output directory `dist`;
   - проверить, нужен ли rewrite fallback на `index.html`.

### Финальные проверки

```bash
npm run test
npm run build
```

Опционально, если нужна проверка реального MOEX API и есть сеть:

```bash
npm run test:integration
```

### Критерий готовности

- UI соответствует направлению из `DESIGN.md`.
- Интерфейс выглядит как строгий финансовый инструмент, а не лендинг или черновик.
- Мобильные ширины проверены.
- Нет горизонтального скролла и визуальных наложений.
- Production build готов.
- README описывает запуск и выбранный путь static deploy.
