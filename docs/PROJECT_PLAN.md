# План проекта: калькулятор российских облигаций

## Цель

Сделать онлайн-калькулятор российских облигаций, который полностью работает в браузере без собственного backend. Данные берем из MOEX ISS API, бизнес-логику пишем на TypeScript.

Основной сценарий использования - мобильный экран. Интерфейс должен быть современным, строгим и рабочим, без ощущения лендинга.

## Экраны

1. Поиск облигаций:
   - строка поиска;
   - список найденных облигаций;
   - краткая информация по каждой бумаге;
   - переход в калькулятор.

2. Калькулятор:
   - параметры бумаги;
   - параметры покупки;
   - режим расчета до погашения, до оферты или до продажи;
   - результаты расчета.

Скриншоты с примерным набором полей лежат в `raw/search.png` и `raw/calc.png`. Look and feel можно сделать лучше, но состав информации стоит учитывать.

## Предварительный стек

- Vite + React + TypeScript.
- TanStack Query для запросов, кеша и состояний загрузки/ошибок.
- React Router для двух экранов.
- Tailwind CSS для mobile-first интерфейса.
- Zod для проверки и нормализации ответов MOEX.
- decimal.js для денежных расчетов.
- Vitest для тестов логики.
- Playwright позже для проверки мобильного UI.

Решение должно оставаться статическим SPA, пригодным для GitHub Pages, Cloudflare Pages, Vercel static hosting или похожего деплоя.

## Договоренности по данным MOEX

### `secid` и `isin`

Внутренний ключ приложения - `secid`.

Пример:

- `secid`: `SU26233RMFS5`
- `isin`: `RU000A101F94`

`isin` показываем пользователю и храним как атрибут, но маршруты и запросы к MOEX строим вокруг `secid`.

### Поиск

MOEX ISS API начинает нормально искать облигации, когда в строке поиска минимум 3 символа.

В UI не нужно отправлять поисковый запрос раньше этого порога. Вместо этого стоит показывать спокойное пустое состояние или подсказку, что нужно ввести минимум 3 символа.

### Board

Облигация может торговаться на нескольких board. Для рыночных данных нужно явно выбрать board.

Базовое правило выбора:

1. `primary_boardid` из поиска MOEX.
2. `marketprice_boardid`, если основная board не найдена в деталях.
3. Fallback по whitelist/priority list, если MOEX не дал ожидаемые данные.

Whitelist нужен как страховка, а не как основной механизм.

### Оферта

`securities.OFFERDATE` считаем ближайшей офертой, если поле заполнено.

Для первого релиза:

- если `OFFERDATE` есть, показываем режим расчета "до оферты";
- дата выхода для этого режима равна `OFFERDATE`;
- цену оферты пытаемся найти в `bondization.offers`;
- если цену найти не удалось, используем 100% от текущего номинала и явно отражаем это в результате.

Расписание оферт полезно хранить в модели, даже если в первом UI не показываем его полностью.

## Предварительная доменная модель

```ts
type BondDetails = {
  secid: string;
  isin: string | null;
  shortName: string;
  name: string;
  boardId: string;
  maturityDate: LocalDate | null;
  nextOfferDate: LocalDate | null;
  offerSchedule: BondOfferScheduleItem[];
};

type BondOfferScheduleItem = {
  date: LocalDate;
  pricePercent: number | null;
  value: number | null;
  type: string | null;
};

type ExitScenario =
  | { kind: "maturity"; date: LocalDate }
  | { kind: "offer"; date: LocalDate; pricePercent: number; priceSource: "moex" | "assumed" }
  | { kind: "sale"; date: LocalDate; pricePercent: number };
```

`LocalDate` - будущий отдельный тип/alias для даты без времени. Внутри приложения даты лучше хранить в нормализованном виде, а форматировать в `dd.MM.yyyy` только в UI.

## Правила процесса

- Делаем строго один этап за раз.
- После каждого этапа останавливаемся.
- Следующий этап начинается только после подтверждения пользователя и коммита.
- Агент не делает коммит сам, если пользователь прямо не попросил.
- Используем принцип KISS: код должен быть простым, читаемым и без неоднозначных "заделов на будущее".
- Не добавляем абстракции "на всякий случай". Новая сущность или слой нужны только если они прямо упрощают текущую задачу или уже естественно следуют из данных.
- В конце этапа агент сообщает:
  - что изменилось;
  - как запустить;
  - какие проверки выполнены;
  - что стоит посмотреть пользователю;
  - suggested commit message.
- В коде должны быть короткие полезные комментарии там, где TypeScript, React или финансовая логика могут быть неочевидны.
- Логику, особенно финансовую, держим отдельно от React-компонентов и покрываем тестами.

## Тесты

Тесты нужны с самого начала.

Обязательные уровни:

- unit-тесты для чистой логики;
- unit-тесты для нормализации MOEX-ответов;
- integration-тесты с реальным MOEX API, запускаемые отдельно.

Обычные тесты не должны зависеть от сети. Проверки реального MOEX API нужны, потому что внешний API может меняться и это важно заметить до поломки приложения.
