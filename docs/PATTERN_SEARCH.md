# Поиск облигаций по glob-паттерну

## Статус

Документ фиксирует решения для последующей реализации. Код поиска и зависимости пока не изменяются.

## Цель

Расширить существующий поиск по `SECID`, `ISIN` и краткому названию облигации простыми glob-паттернами, не усложняя обычный поиск и не предоставляя пользователю произвольные регулярные выражения.

## Формат запроса

- Обычный текст сохраняет текущее поведение: ищется подстрока.
- `*` соответствует нулю или более произвольным символам.
- `?` соответствует ровно одному произвольному символу.
- `\` экранирует следующий служебный символ: `\*`, `\?`, `\\`.
- Регистр не учитывается.
- Glob-паттерн сопоставляется с подстрокой значения поля. То есть поиск уже неявно допускает произвольные символы до и после паттерна.
- Один паттерн независимо проверяется по `SECID`, `ISIN` и `shortname`; совпадения хотя бы по одному полю достаточно.

Примеры:

| Запрос | Значение |
| --- | --- |
| `26233` | Обычный поиск подстроки по всем трём полям |
| `SU262*` | Значение поля содержит фрагмент, начинающийся с `SU262` |
| `*сек` | Значение поля содержит `сек`; завершающая `*` не нужна |
| `RU000A10????` | Значение поля содержит `RU000A10`, после которого находятся ровно четыре символа |
| `*серия \**` | Значение поля содержит `серия *`; `*` после обратной косой черты считается обычным символом |

Паттерн должен содержать минимум три обычных символа. Неэкранированные `*` и `?`, а также сама обратная косая черта в это количество не входят. Это не позволяет запросам наподобие `*` или `???` выбирать весь снимок облигаций.

## Библиотека

Для сопоставления использовать [`wildcard-match`](https://github.com/axtgr/wildcard-match).

Причины выбора:

- библиотека поддерживает необходимый минимальный синтаксис: `*`, `?` и экранирование;
- умеет работать с произвольными строками, а не только с файловыми путями;
- поставляется со встроенными TypeScript-типами;
- работает в браузере и имеет ESM-сборку;
- не имеет runtime-зависимостей;
- паттерн компилируется один раз в функцию сопоставления.

При создании matcher обязательно отключить разделители:

```ts
import wildcardMatch from "wildcard-match";

const matches = wildcardMatch(`*${normalizedPattern}*`, {
  separator: false,
});
```

`separator: false` заставляет библиотеку считать `/` обычным символом. Без этой настройки `*` и `?` ведут себя как wildcards файлового пути и не пересекают разделители.

Для нечувствительности к регистру предпочтительно сохранить существующую нормализацию через `toLocaleUpperCase("ru-RU")` для паттерна и проверяемых значений. Альтернатива — передать библиотеке `flags: "iu"`, но смешивать оба подхода не нужно.

## Предполагаемая схема реализации

1. Обрезать пробелы по краям запроса.
2. Определить, содержит ли запрос glob-синтаксис.
3. Проверить минимальное количество обычных символов и корректность экранирования.
4. Для обычного текста оставить текущий поиск подстроки.
5. Для glob-запроса нормализовать, неявно обернуть паттерн в `*…*` и скомпилировать matcher один раз.
6. Проверить matcher по нормализованным `secid`, `isin` и `shortname` каждой облигации.
7. Сохранить существующие сортировку результатов и ограничение их количества.

Matcher нельзя компилировать внутри `.filter()` или отдельно для каждого поля: компиляция должна выполняться один раз на поисковый запрос.

Рекомендуется также ограничить максимальную длину запроса разумным UI-лимитом. Конкретное значение следует выбрать при реализации.

## Что не входит в первую версию

- префиксы полей (`secid:`, `isin:`, `name:`);
- регулярные выражения;
- логические операторы `AND`, `OR`, отрицание и скобки;
- диапазоны символов (`[0-9]`), brace expansion и extglob;
- несколько паттернов в одном запросе.

Префиксы полей можно добавить позднее без изменения базовой glob-семантики.

## Минимальный набор тестов

- обычный текст продолжает искать подстроку без учёта регистра;
- `SU262*` совпадает с `SECID`, содержащим фрагмент с этим префиксом;
- `*Газпром*` совпадает с частью `shortname` без учёта регистра;
- `?` соответствует ровно одному символу;
- glob сопоставляется с произвольной подстрокой поля, в том числе когда `*` указан только с одной стороны;
- `\*` и `\?` интерпретируются как обычные символы;
- `/` может быть поглощён символами `*` и `?` при `separator: false`;
- паттерны с менее чем тремя обычными символами не запускают поиск;
- matcher компилируется один раз на запрос и применяется ко всем трём полям;
- обычная сортировка и лимит результатов не меняются.
